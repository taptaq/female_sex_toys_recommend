# JTBD 优化实施路线图

> 基于 `docs/2026-05-03-jtbd-optimization-task-list.md` 继续向下拆分，目标是把产品判断转成可以直接排期和开工的实施清单。

日期：2026-05-03

---

## 目标

把当前产品从“有很多功能的推荐工具”，继续收束成“帮助用户更安心下决定的选择向导”。

这一轮不追求加更多炫功能，而是围绕三个核心目标推进：

1. 降低认知负担
2. 缩短从匹配到购买决策的距离
3. 提高推荐结果的信任感与可解释性

---

## 实施原则

### 1. 优先改主流程，不先改边缘模块

优先级顺序：

1. 问卷页
2. 结果页
3. 模型路由与服务端推荐链路
4. 档案页
5. 首页文案与入口收束

### 2. 优先做“减少犹豫”的能力

如果一个改动只是增加信息密度、增加可选项、增加炫酷度，但不能明显减少用户犹豫，就不应排在前面。

### 3. 普通用户不应该理解系统内部机制

包括但不限于：

- 模型 provider
- 内部字段
- 中间评分
- 路由策略

这些都应保留在后台逻辑与开发日志，不暴露在用户主界面。

## 阶段拆分

## Phase 1：收束结果页，补足决策承接

### 目标

先把结果页从“推荐展示页”提升成“决策落地页”。

### 主要任务

#### 1. 隐藏前台模型切换，只保留“重新生成推荐”

**涉及文件**

- `src/pages/ResultsPage.tsx`
- `src/lib/result-models.ts`
- `src/lib/result-recalibration.ts`
- `src/App.tsx`
- `src/pages/ResultsPage.test.tsx`

**要做什么**

- 去掉结果页模型选择 UI
- 去掉用户可见的模型来源展示
- 保留单一按钮：`重新生成推荐`
- 按钮语义从“换模型试试”改为“给我再来一版更合适的”

**完成标准**

- 用户看不到 provider 相关概念
- 结果页更像决策页，而不是调试面板

---

#### 2. 新增“下一步怎么办”模块

**涉及文件**

- `src/pages/ResultsPage.tsx`
- `src/lib/recommendation-results.ts`
- `src/App.tsx`
- 如需后端增强：`src/server/app-ai-service.ts`

**要做什么**

- 在主推荐下方新增“下一步建议”
- 内容拆成三类：
  - 去哪里买更放心
  - 下单前检查什么
  - 收到货后第一步做什么

**推荐实现顺序**

先做本地规则版本，再考虑 AI 丰富文案：

- 本地规则先根据 `静音 / 防水 / 新手 / 同住 / 情侣 / 隐蔽` 生成建议
- 后续如需要更自然表达，再接 AI enhancement

**完成标准**

- 用户拿到推荐后，不会停在“所以我下一步要干嘛”

---

#### 3. 参数 chip 接入知识解释

**涉及文件**

- `src/pages/ResultsPage.tsx`
- `src/data/knowledge-nebula.ts`
- `src/lib/knowledge-nebula-route.ts`
- `src/App.tsx`
- 如需轻提示：新增 `src/components/...`

**要做什么**

- 给 `maxDb / waterproof / motorType` 等 chip 增加解释入口
- 支持两种形式二选一：
  - tooltip 弹层
  - 跳到对应知识主题 / 卡片

**推荐优先级**

先做轻解释，再做深跳转：

1. 先 tooltip
2. 再接知识星云详情页

**完成标准**

- 新手在结果页就能理解关键参数

---

#### 4. 增加“为什么不推荐”模块

**涉及文件**

- `src/pages/ResultsPage.tsx`
- `src/lib/recommendation-results.ts`
- `src/lib/quiz-branching.ts`
- `src/pages/ResultsPage.test.tsx`

**要做什么**

- 在主推荐区域附近新增 1-2 条“不建议优先看什么”
- 规则优先根据答题条件生成

**示例逻辑**

- 同住 + 静音敏感：不建议高噪音路线
- 新手 + 不确定：不建议强刺激路线
- 情侣共玩 + 边界敏感：不建议控制复杂路线

**完成标准**

- 结果页不仅会推荐，也会明确边界

---

## Phase 2：优化问卷，减少用户卡顿

### 目标

让用户在“不知道”的情况下也能继续顺利完成匹配。

### 主要任务

#### 5. 关键题加入“我不确定 / 都可以 / 帮我判断”

**涉及文件**

- 问卷题目定义文件
- 问卷分支逻辑文件
- `src/App.tsx`
- 问卷相关测试文件

当前代码里需要重点排查：

- `src/lib/quiz-branching.ts`
- 题目数据来源文件
- 已有的分支计划文档对应实现位置

**要做什么**

- 在关键题新增不确定选项
- 为这些选项定义保守默认分支
- 避免结果完全失真

**优先题目**

- 刺激路线
- 强度偏好
- 外部 / 入体 / 复合
- 偏隐蔽还是偏体验

**完成标准**

- 用户不需要“硬选一个并不懂的答案”

---

#### 6. 增加“改单项 / 上一步修正”能力

**涉及文件**

- `src/App.tsx`
- 问卷页面组件
- 结果页微调逻辑
- `src/lib/result-tuning.ts`

**要做什么**

- 在结果页提供回改关键条件入口
- 或在问卷页明确支持返回上一步修改

**推荐做法**

先做最轻版本：

- 结果页提供“改预算 / 改静音 / 改场景”三个快捷返改入口

后续再决定是否做完整的“按题回退”

**完成标准**

- 用户不必为一个条件的犹豫而重做整套问卷

---

## Phase 3：后台自动模型路由

### 目标

让“重新生成推荐”成为真正面向用户的产品动作，而不是模型选择动作。

### 主要任务

#### 7. 重构结果重校准请求结构

**涉及文件**

- `src/lib/result-recalibration.ts`
- `src/App.tsx`
- `src/server/app-ai-recalibration-route.ts`
- `src/server/app-ai-recalibration-route.test.ts`

**要做什么**

- 前端不再传用户选择的 `targetProvider`
- 改成传：
  - `strategy: "auto"`
  - 或不传 provider，由服务端自行决定

**完成标准**

- 前端和用户彻底解除对 provider 的感知

---

#### 8. 服务端新增自动路由策略

**涉及文件**

- `src/server/app-ai-service.ts`
- `src/lib/app-ai-chain.ts`
- `src/server/app-ai-service.test.ts`
- 如需新模块：`src/server/result-provider-router.ts`

**要做什么**

- 新增 `resolveRecalibrationPlan()` 或类似能力
- 输出：
  - `rerankProvider`
  - `enhancementProvider`
  - `fallbackOrder`

**推荐策略**

- 首次默认：`dmxapi-mimo`
- 若用户二次重生成，或结果质量偏弱：
  - 升级到 `dmxapi-minimax` / `dmxapi-qwen`
- 失败后按 provider ladder 自动 fallback

**完成标准**

- 用户点击“重新生成推荐”时，后台可自动决定继续当前 provider 或升级

---

#### 9. 按任务拆分 provider 使用

**涉及文件**

- `src/server/app-ai-service.ts`
- 对应测试文件

**要做什么**

- `Top3 重排` 与 `结果说明增强` 不再强制用同一个 provider
- 重排优先稳定
- 文案增强优先自然表达

**完成标准**

- 推荐链路更像“产品策略”，而不是“一个模型跑完全流程”

---

## Phase 4：档案页从存档变成复盘工具

### 目标

让“我的装备匹配档案”更像复盘中心，而不是数据回显页。

### 主要任务

#### 10. 档案页增强自然语言摘要

**涉及文件**

- `src/pages/ProfilesPage.tsx`
- `src/lib/user-recommendation-profile.ts`
- `src/pages/ProfilesPage.test.tsx`

**要做什么**

- 生成每份档案的“当时为什么推荐”
- 弱化字段清单的中心位置
- 提升“当时更在意什么”的可读性

**完成标准**

- 用户能快速回忆自己的决策背景

---

#### 11. 支持候选收藏 / 稍后比较（已落地轻量版）

**涉及文件**

- 结果页
- 档案页
- 用户推荐档案 API
- 数据库存储结构

**可能新增表 / 字段**

- `saved_candidate_ids`
- 单独的 favorites 表

**推荐方式**

优先用附加字段扩展当前档案结构，而不是先新建复杂子系统。

**完成标准**

- 用户可以保存 2-3 个候选继续比较，而不只是保存整份结果
- 当前实现已采用档案 payload 附加字段：`savedCandidateIds` 与 `savedCandidates`
- 结果页支持“稍后比较”候选选择，档案详情支持回看候选
- 独立 favorites 表暂缓，等出现跨档案长期收藏需求后再升级

## Phase 5：首页继续收束

### 目标

让首页继续保持“一个主任务 + 两个次级入口”的清晰结构。

### 主要任务

#### 12. 合并首页隐私说明

**涉及文件**

- `src/pages/HomePage.tsx`
- `src/pages/HomePage.test.tsx`

**要做什么**

- 去掉重复 chip
- 将隐私能力合并到登录区域的一句主文案中

**完成标准**

- 首页信息层级更轻

---

#### 13. 登录入口继续弱化“先登录”感

**涉及文件**

- `src/pages/HomePage.tsx`
- `src/components/AuthPanel.tsx`
- 结果页保存档案相关交互

**要做什么**

- 首页登录入口继续保持次级
- 真正的强触发点放在：
  - 保存档案
  - 查看我的档案

**完成标准**

- 首页不会让用户产生“要先注册才能用”的压迫感

## 数据与接口层补充

### 可能需要新增或调整的数据结构

可选字段：

- `summary_reason`
- `primary_route_summary`
- `decision_context_summary`

## 推荐开发顺序

### 第一轮

1. 结果页隐藏模型切换
2. 结果页新增“下一步建议”
3. 结果页参数解释桥接知识星云

### 第二轮

1. 问卷增加“不确定”选项
2. 结果页增加“为什么不推荐”
3. 增加改单项能力

### 第三轮

1. 自动模型路由
2. rerank / enhancement provider 拆分

### 第四轮

1. 档案页自然语言复盘
2. 收藏候选 / 稍后比较

### 第五轮

1. 首页隐私表达收束
2. 登录入口进一步弱化

### 并行专项

1. 知识星云 3D 渲染性能治理
2. 首页星云与详情驾驶舱的桌面 / 移动端性能预算分层
3. 长时间停留、切后台、反复切换主题时的降频与资源回收校验

---

## 最终落点

做完这一轮之后，产品理想上会变成这样：

- 首页不再像入口集合页，而是明确把用户推向“开始匹配”
- 问卷不再逼用户在不懂的时候瞎选
- 结果页不只是展示推荐，而是帮助用户继续做决定
- 知识星云不再独立悬空，而是变成决策过程里的解释层
- 模型能力退到幕后，用户只感知到“系统在帮我给出更合适的一版”
- 档案页从数据存档升级为决策复盘工具
