# 知识星云银河母云刷新 Spec

日期：2026-04-30

## 背景

当前 `/knowledge` 首页虽然已经完成了从碎片卡片到 3D 星云 Hub 的第一次改造，但用户对现阶段视觉效果仍然明确不满意，核心反馈是“太丑”“不像雾团”“更像几团粒子球”。用户给出的参考图更接近真实深空星云：中心高亮、外围大面积雾带、星尘遍布、整体像一整片银河母云，而不是 5 个独立发光对象并排陈列。

因此，这次不是继续微调粒子数量、颜色或 hover 特效，而是重新定义首页 Hub 的视觉母体：从“5 团独立主题云”转为“1 整片银河母云，内部显形出 5 个主题亮区”。

## 本次目标

- 把首页 Hub 的第一观感改成真实深空母星云，而不是几个独立云球
- 保留现有 5 个主题入口与详情页路由结构
- 让 5 个主题在视觉上表现为母云中的 `5 个亮度凝结区`
- 让文字层从卡片感明显的 UI 组件，改成轻薄、浮在雾里的标题铭牌
- 保留当前已确认的交互骨架：
  - 先聚合成母云
  - 再从母云中分化出 5 个主题区域
  - 点击后镜头推进，再进入详情页
- 同时兼容 PC 与移动端，不依赖 hover 才能理解页面

## 已确认的用户决策

- 整体方向：`A. 银河母云`
- 5 个主题不是 5 团独立球体，而是整片母云中的 5 个局部亮区
- 标题处理选择：保留极轻的可点击容器，但要尽量弱化卡片感
- 动效基调：保留镜头推进，但视觉重点从“球体聚焦”改成“穿入局部星云”
- 视觉参考：高亮白紫核心、蓝紫雾带、深空星点、大范围铺开的云气

## 核心体验定义

用户进入 `/knowledge` 时，应该感觉自己在看一整片漂浮于深空中的银河母云。

这片母云会先从中心聚亮，再带动外围雾带缓慢呼吸展开。随后，母云内部的 5 个区域不是“裂开成 5 团规则云块”，而是像不同位置逐渐出现亮度重心、星尘密度和局部雾核，让人意识到这里藏着 5 个可进入的主题区。

用户的注意力应该先被整片星云吸住，再逐渐识别出 5 个主题，而不是第一眼就看到 5 个整齐摆放的入口按钮。

## 视觉构图

### 1. 母云主体

母云主体是整个页面的视觉核心，应该覆盖当前可视区域的大部分宽度，而不是只占中间一块。

画面特征：

- 中央有一个高亮云核，接近白色偏紫
- 外围是多层蓝紫、冷粉、冷白混合的半透明雾带
- 雾带整体带有轻微旋臂和流向，但不做夸张的科幻能量圈
- 星点与星尘分布在前景、中景、远景，不是只在背景平铺一层
- 整体亮度中心明确，但四周仍有持续延展的云气，避免中心一坨、边缘空掉

### 2. 五个主题亮区

5 个主题在视觉上不再被表现为 5 个独立、轮廓明确的云团，而是 5 个局部增强区：

- 某一片雾更亮
- 某一片雾更浓
- 某一片星尘旋流更活跃
- 某一片有更明显的局部发光核心

它们应该“从母云里长出来”，而不是“摆在母云前面”。

### 3. 标题层

标题仍需默认可见，但要从当前相对明显的卡片盒子感，改成更轻、更薄、更像浮在云里的铭牌。

目标特征：

- 仍保留足够大的点击热区
- 边框更弱，背景更淡
- 更多依赖柔光、模糊、透明度来托住文字，而不是块状面板
- 标题像悬在云雾上的导航标记，而不是一张功能卡片
- 移动端默认透明度比桌面端稍高，保证无 hover 时仍易读

## 动画与状态节奏

沿用现有四段状态机，但重新定义它们的视觉表现。

### `aggregate`

首页初始先显示中央母云聚亮。

表现：

- 中心云核亮度上升
- 外围雾层轻微呼吸展开
- 星尘围绕中心低速流动
- 此阶段重点是“整片云正在苏醒”，不是“有个球在闪”

### `split`

这里的“split”不再是视觉上硬拆成 5 团，而是母云内部 5 个区域开始显形。

表现：

- 5 个主题区逐个出现局部亮度增强
- 局部雾核与星尘密度升高
- 不同区域错峰显形，像从母云里被识别出来
- 仍保持整片母云连续，不出现强边界切割

### `idle`

进入 idle 后，整片母云仍然存在，5 个主题亮区只是其中更活跃的区域。

表现：

- 前景星尘、中景雾层、远景星点持续低频漂移
- 各主题亮区缓慢脉动，但不能统一节奏
- 云层边界松散、不规则，避免统一椭圆形

### `focus`

点击某个主题时，视觉焦点是“穿向那片局部星云”，而不是“镜头冲向一个规则球体”。

表现：

- 目标区域变得更亮、更通透
- 周围雾带轻微旋开或拉伸
- 其他区域退为后景，存在感下降
- 相机向目标亮区推进，再切入详情页

## 页面结构与组件职责

当前组件边界可以保留，但每层职责需要偏向“整片母云”而不是“多个云球”。

### `NebulaScene3D`

从“云球容器渲染器”升级为真正的母云场景层，负责：

- 母云底层
- 大面积雾带与旋臂
- 前后景星尘
- 5 个主题亮区的空间分布
- 相机推进时的整体景别变化

### `NebulaCluster`

不再代表一个完整独立云球，而是某个主题亮区的局部增强器，负责：

- 局部亮度增强
- 局部星尘旋流
- 局部雾核密度变化
- hover / focus 时的局部响应

它的输出应更不规则、更松散，不允许所有主题亮区都长成相似椭圆。

### `NebulaLabelLayer`

保留 DOM 层来保证可读性与点击热区，但样式上要进一步削弱卡片感，强化“薄雾铭牌”感。

### `KnowledgeNebulaField`

继续作为 Hub 编排层，保留：

- viewport 适配
- aggregate / split / idle / focus 状态编排
- 相机推进后的延迟路由切换

但它需要适配新的视觉模型，不再把每个主题当成一团完整云体去驱动。

## 技术策略

### 保留现有技术栈

继续使用：

- `three`
- `@react-three/fiber`
- `@react-three/drei`
- `motion/react`

不额外引入更重的体积云渲染库，优先在当前栈内完成更高级的母云质感。

### 素材与特效库边界

用户已确认：如果实现高质感星云需要相关素材，可以自动下载；也可以参考或引入其他特效库。

执行时采用以下优先级：

1. 先使用程序化纹理和现有 Three/R3F 能力生成高质量母云，避免素材版权、体积和加载稳定性风险。
2. 如果程序化雾场仍无法达到参考图的雾化质感，可下载可商用或公有领域的星云/烟雾/星尘纹理，并将来源记录在项目文档或资源旁注中。
3. 如需新增特效库，优先选择体积小、维护活跃、能与当前 Vite + React + R3F 栈直接兼容的库。
4. 不引入会显著拖慢移动端首屏、需要复杂后处理链、或授权不清晰的素材与库。

### 视觉实现原则

为了更接近参考图，第一优先级不是增加更多离散粒子，而是增加“大面积、柔软、分层的雾场”。

实现上应优先考虑：

- 多层大尺寸软雾 sprite / plane 叠加
- 不同尺度的云带和 halo
- 前中后景星尘分层
- 更强的整体色场，而不是单个对象边缘发光
- 每个主题区使用不规则局部增强，而不是统一模板复制

## PC 与移动端策略

### PC

- 保留更完整的大雾场层次
- 鼠标移动时允许轻微视差
- hover 时目标亮区通透度与标题清晰度提升

### 移动端

- 仍然呈现完整母云，不可退回成简化的几个点
- 默认标题更清晰
- 降低粒子与动态扰动密度
- 缩小动效幅度，避免炫动感过强

## 非目标

本次不包含：

- 改写知识数据结构
- 改写详情页内容模块
- 引入复杂后处理链或高成本体积 raymarching
- 做完全写实的天文模拟
- 自动提交代码或文档

## 成功标准

如果这次刷新成功，用户进入首页时应当立即感受到：

- 这是一整片银河母云，而不是几个发光球
- 5 个主题是从母云里浮现出来的，不是摆在前面的按钮
- 标题可读，但视觉上已经不是“卡片”
- 页面四周也有星云延展，不会只在中间挤成一坨
- 点击某个主题时，像穿进局部星云，而不是切到一个普通详情页

## 实施建议

1. 先重做 `NebulaScene3D` 的母云底层与大雾场结构
2. 再重写 `NebulaCluster`，让其只负责局部亮区增强
3. 然后瘦身 `NebulaLabelLayer` 的 UI 盒子感
4. 最后微调 `KnowledgeNebulaField` 的文案、过渡和移动端可读性
