# 身体人格测试与隐藏偏好识别设计

## 目标

在现有推荐结果链路中增加一个轻量但有差异化价值的 `身体人格测试` 功能，用来回答用户“我长期更适合哪类装备路线”，并进一步补一层 `隐藏偏好识别`，让用户不仅得到这次场景匹配结果，还能得到一份更私密、更可解释、更容易付费解锁的长期画像。

本功能的核心目标不是做娱乐测试，而是让用户：

- 更快被理解
- 更安心地做选择
- 更有代入感地获得专属装备路线

## JTBD 结论

用户真正“雇佣”这个功能，不是为了做题本身，而是为了在敏感、模糊、容易犹豫的消费决策里，获得一种“终于有人懂我，而且顺手帮我缩小选择范围”的体验。

### Functional Jobs

- 判断自己长期更适合哪类体验路线
- 区分“这次场景想要什么”和“我通常适合什么”
- 在安全感、隐私、刺激节奏、感官层次、关系联动之间找到平衡
- 更快缩小候选范围，减少踩雷
- 更自然地识别自己是否偏向隐藏款、伪装款或低存在感路线

### Social Jobs

- 不想显得自己很外行
- 不想在选择过程中暴露尴尬或不确定感
- 希望自己的偏好被正常理解，而不是被评判
- 对隐藏款用户来说，希望自己像一个会选、会藏、会拿捏的人

### Emotional Jobs

- 被识别和命名的爽感
- 从信息过载中被解救出来的轻松感
- 在敏感消费中获得安全感、私密感和一点点刺激感
- 拿到一份“只有我才会看到”的专属答案

### 关键痛点

- 用户不知道自己真正适合什么，只知道自己不想要什么
- 普通推荐容易像通用榜单，缺少“只属于我”的感觉
- 隐私、伪装、同住、收纳压力等需求很难被普通推荐表达清楚
- 很多测试只能给出人格名字，不能直接转成行动和购买建议

### 核心爽点

- `身份爽点`：我原来是这种类型
- `隐秘爽点`：我有自己偏好的隐藏路线
- `专属爽点`：这份结果是沿着我的答题链路推出来的
- `掌控爽点`：我不只被测试，我还拿到具体可执行的路线

## 范围

本次设计包含：

- 结果页新增 `身体人格测试` 解锁卡
- 新增独立测试流程
- 6 种身体人格原型
- 隐藏偏好识别结果
- 免费结果层与 `0.5 元` 解锁层
- 推荐结果联动逻辑
- 登录后保存结果
- 订单、解锁与权限模型
- 可回收的训练与校准数据沉淀

本次设计不包含：

- 真正接入正式微信/支付宝商户
- 后台运营管理页
- 完整的历史订单中心
- 独立的人格社区、分享海报、排行榜
- 用模型直接替代现有主推荐算法

## 已确认方向

基于当前讨论，已确认如下产品方向：

- 入口放在 `结果页解锁卡`
- 付费结果内容为 `画像 + 产品清单`
- 单次价格为 `0.5 元/次`
- 解锁绑定到 `这一次人格测试结果`
- 题目来源采用 `独立新题集`
- 第一版人格原型数为 `6 种`
- 判型方式采用 `规则判型 + AI 润色`
- 名称风格采用 `混合命名`
- 免费层展示 `人格名称 + 基础画像 + 少量方向提示`
- 登录后保存到账户
- 支付流形态预留 `移动端微信 H5 / PC 端微信扫码`，同时保留后续支付宝接入空间

## 用户体验

### 结果页入口

身体人格测试不抢主推荐动作，位置放在：

- 主推荐结果之后
- 简短匹配解释之后
- 备选推荐与知识补充之前

入口卡片的用户理解应是：

- 这次推荐回答“你现在想要什么”
- 身体人格测试回答“你长期更适合什么路线”

入口文案方向：

- `测一测你的身体人格，看看你长期更适合哪类装备路线`
- `这次推荐是场景匹配，你的长期偏好可能是另一条线`

### 测试流程

进入测试后，控制在 `6-8 题`，每题单选，节奏尽量快，不做长问卷。

每道题必须映射到明确维度，禁止出现仅用于装饰、但不参与结果计算的选项。

### 免费结果层

用户完成测试后，先看到免费结果，不强制付费后才知道自己是谁。免费结果包含：

- 人格名称
- 一句基础画像
- `1-2` 条方向提示
- 一条“为什么是这个类型”的短解释
- 一条隐藏路线倾向
- 解锁完整结果入口

免费层的目标是先建立“有点准”的信任，而不是先做拦截。

### 付费解锁层

支付成功后展开完整结果，内容包含：

- 完整人格画像
- 更适合的装备路线
- 暂不优先的路线提醒
- 隐藏偏好识别结果
- `3-5` 个更贴合人格的产品清单
- 人格结果与本次场景推荐的联动解释
- 经过 AI 润色的个性化建议文案

## 人格原型

第一版采用以下 6 种原型：

1. `微光型·慢热探索者`
2. `星幕型·隐秘安全感者`
3. `潮汐型·感官共振者`
4. `彗火型·即时点燃者`
5. `星环型·节奏掌控者`
6. `双轨型·关系联动者`

这些原型不直接等同于性别，而是作为更上层的长期偏好框架使用。

## 判型维度

第一版固定为 6 条核心维度：

- `安全感需求`
- `隐私与伪装需求`
- `刺激建立速度`
- `感官层次偏好`
- `掌控欲与操作偏好`
- `关系联动倾向`

每道题至少命中 `1-2` 条维度，并写入权重，保证所有选择链路都参与最终判型。

## 判型逻辑

采用 `规则判型 + AI 润色` 两层结构。

### 第一层：规则判型

规则层负责：

- 计算维度分数
- 输出候选人格
- 计算主人格与次人格
- 给出可解释原因
- 生成隐藏偏好初判

规则层必须稳定、可回放、可测试，不能依赖模型才有结果。

### 第二层：AI 润色

AI 层只负责：

- 把已确定的人格结果润色成更自然的中文表达
- 生成更贴合用户链路的解释文案
- 串联人格结果与推荐结果的关系

AI 不负责决定人格类型本身，也不允许覆盖规则层硬边界。

## 隐藏偏好识别

隐藏偏好不是一个简单布尔值，而是一个独立结果模块。

### 主类型

第一版使用 4 个最容易理解的主类型：

- `零识别型`
- `日常器物型`
- `美妆伪装型`
- `口袋随身型`

### 能力标签

再补两个附加指标：

- `隐藏力`：`S / A / B`
- `共居安心度`：`高 / 中 / 低`

隐藏偏好结果会同时用于：

- 人格测试付费结果页展示
- 后续产品清单排序
- 推荐解释增强

## 与现有推荐链路的关系

身体人格测试不是替代现有推荐问答，而是上层“长期校准器”。

推荐系统职责拆分为：

- 现有场景答题：回答“你这次想要什么”
- 身体人格测试：回答“你通常适合什么路线”
- 最终联动结果：在满足硬条件后做软排序增强

### 硬条件优先

以下约束不能被人格测试覆盖：

- 性别向约束
- 用户明确排除的刺激方式
- 预算上限
- 隐私与伪装硬要求
- 非玩具类排除规则

### 软排序增强

人格结果主要影响以下排序方向：

- 温和渐进 vs 直接点燃
- 低存在感 vs 更强调体验表达
- 单人自我探索 vs 关系联动
- 更重层次感 vs 更重效率反馈
- 更适合的隐藏路线

## 数据模型

建议新增 3 张表。

### 1. `public.body_persona_sessions`

保存一次测试会话与免费结果。

建议字段：

- `id uuid primary key default gen_random_uuid()`
- `recommendation_session_id text`
- `user_id uuid references auth.users(id) on delete set null`
- `source_page_route text not null default '/results'`
- `question_version text not null`
- `scoring_version text not null`
- `answers jsonb not null default '{}'::jsonb`
- `answer_path jsonb not null default '[]'::jsonb`
- `dimension_scores jsonb not null default '{}'::jsonb`
- `primary_persona_code text not null`
- `secondary_persona_code text`
- `hidden_route_code text`
- `hidden_power_grade text`
- `co_living_comfort_grade text`
- `free_summary jsonb not null default '{}'::jsonb`
- `full_report jsonb`
- `status text not null default 'completed_free'`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

### 2. `public.body_persona_unlock_orders`

保存一次付费解锁订单。

建议字段：

- `id uuid primary key default gen_random_uuid()`
- `persona_session_id uuid not null references public.body_persona_sessions(id) on delete cascade`
- `user_id uuid references auth.users(id) on delete set null`
- `amount_cent integer not null default 50`
- `currency text not null default 'CNY'`
- `channel text`
- `merchant_order_no text unique`
- `payment_provider text`
- `status text not null default 'pending'`
- `paid_at timestamptz`
- `expired_at timestamptz`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

### 3. `public.body_persona_unlock_entitlements`

保存一次结果是否已经被解锁，方便快速鉴权。

建议字段：

- `id uuid primary key default gen_random_uuid()`
- `persona_session_id uuid not null unique references public.body_persona_sessions(id) on delete cascade`
- `order_id uuid references public.body_persona_unlock_orders(id) on delete set null`
- `user_id uuid references auth.users(id) on delete set null`
- `unlocked_scope text not null default 'full_report'`
- `created_at timestamptz not null default now()`

## 状态流转

测试与解锁流程建议采用以下状态：

1. 用户完成测试，创建 `body_persona_session`
2. 生成免费结果，状态为 `completed_free`
3. 用户点击解锁，创建 `pending` 订单
4. 支付成功后写入 entitlement
5. session 状态更新为 `unlocked`
6. 登录用户可在档案体系中查看已解锁结果

如果支付失败或超时：

- session 保留免费结果
- 订单状态改为 `failed` 或 `expired`
- 用户可再次发起新的解锁订单

## 页面与接口结构

建议采用与现有推荐会话、反馈存储类似的 `store + route + ensure schema` 模式。

### 前端页面

- 结果页解锁卡
- 人格测试页或人格测试弹层
- 免费结果页
- 付费完整结果展开区
- 登录后历史结果查看入口

### 后端接口

第一版建议至少包含：

- `POST /api/body-persona/sessions`
- `GET /api/body-persona/sessions/:id`
- `POST /api/body-persona/orders`
- `POST /api/body-persona/orders/:id/pay-callback` 或等效模拟确认接口
- `GET /api/body-persona/sessions/:id/unlock-status`

如果当前阶段暂不接真实支付，也要保证订单和 entitlement 的状态机先成立。

## 结果解释原则

免费层与付费层都必须能回溯到用户选择链路，解释逻辑建议采用：

- 你在 `隐私需求`、`慢热节奏`、`掌控偏好` 得分更高
- 因此更偏向 `星幕型 / 星环型`
- 这会让你更适合 `低存在感 / 易收纳 / 节奏可控` 的路线

解释文案必须建立在结构化分数之上，不能生成与分数矛盾的内容。

## 推荐产品清单策略

付费后展示的人格产品清单不是重新做一遍主推荐，而是从现有候选池中做二次人格重排，必要时再补少量人格专属候选。

排序时优先考虑：

- 与人格维度的匹配度
- 与隐藏偏好的匹配度
- 与本次场景结果的一致性
- 与用户硬条件的兼容性

禁止推荐：

- 与用户性别向冲突的产品
- 已明确排除的非玩具类
- 与高隐私需求明显冲突的高存在感产品

## 支付策略

第一版按正式支付产品形态设计，但实现上允许先保留支付适配层。

### 当前阶段

- 完成订单、解锁、状态机与前端交互
- 支付通道可先使用模拟成功流程或受控测试通道

### 后续接商户

- 移动端优先微信 H5
- PC 端优先微信扫码
- 预留支付宝能力，但不要求本次实现

## 数据沉淀与训练价值

本功能除了成交转化，还会形成一批高价值训练数据。

建议保存：

- 人格测试完整答题链路
- 免费结果与完整结果
- 本次主推荐快照
- 用户是否解锁
- 用户是否在结果页重新推荐
- 用户是否保存档案
- 人格结果与最终点开的产品

后续可用于：

- 校准人格判型规则
- 优化人格与产品路线的映射
- 识别哪些人格更容易触发重新推荐
- 分析哪些隐藏偏好与哪些装备类型真实转化更高

## 成功标准

第一版成功不以“题很多”或“文案很炫”为标准，而是看以下几个信号：

- 用户能理解人格测试与主推荐的区别
- 免费层能建立“有点准”的感受
- 付费层能给出比普通推荐更强的专属感
- 人格结果能明显增强推荐解释与隐藏路线表达
- 每个答题选项都能进入评分与解释链路

## 测试策略

### 规则与模型层测试

- 维度加权正确
- 同一输入可稳定回放同一结果
- 人格代码与隐藏路线代码映射正确
- AI 润色不允许改写规则层硬结果

### 前端测试

- 结果页存在解锁卡
- 免费结果展示正确
- 未支付时完整结果不可见
- 已支付时完整结果可展开
- 登录与未登录路径行为正确

### 后端测试

- schema ensure 正确创建 3 张新表
- session 可写入与查询
- order 状态流转正确
- entitlement 可正确鉴权
- 非法请求和重复解锁能被安全处理

## 非目标与后续扩展

后续可扩展但本次不强做：

- 次人格更细分的二级标签
- 人格海报分享
- 历史人格变化趋势
- 支付后解锁更丰富的知识解释内容
- 将人格结果并入长期用户档案首页

## 实现原则

- 先把规则骨架和解释链路打稳，再加 AI 润色
- 先做低门槛、快节奏体验，不做长问卷
- 先保证“准和可解释”，再追求“更玄更有戏剧性”
- 先把隐藏偏好做成真实可用的排序信号，而不是装饰性标签
- 先让用户做出更安心的选择，再谈更复杂的商业化延展
