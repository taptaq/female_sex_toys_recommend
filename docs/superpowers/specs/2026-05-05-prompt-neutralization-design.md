# Prompt 中性化设计

## 目标
在不改变现有业务分类、标签体系和测试语义的前提下，清理当前代码里直接发送给模型的 prompt 文本，将高风险和中风险的敏感措辞替换为更中性的业务表达，降低模型安全拦截、拒答或输出跑偏的概率。

本轮重点不是“统一全仓术语”，而是“只收敛 prompt 表层文案”。

## 已确认决策
- 本轮只处理真正发给模型的 prompt 文本，不扩展到整仓业务词表。
- `性健康装备选品专家` 一并替换为更中性的角色描述。
- 不改 `function_tags`、分类命中词、性别判定词、测试样例里的业务语义。
- 不追求一次性清空所有显式业务词；优先处理高风险和中风险项。

## 范围
- 推荐链路 prompt：
  - `src/server/app-ai-service.ts`
  - `src/App.tsx`
- 官方站 OCR / cleaner prompt：
  - `src/scraper/svakom-official/crawler.ts`
  - `src/scraper/svakom-official/cleaner.ts`
  - `src/scraper/lovense-official/cleaner.ts`
  - `src/scraper/nomitang-official/cleaner.ts`
  - `src/scraper/wevibe-official/cleaner.ts`
  - `src/scraper/zalo-official/crawler.ts`
  - `src/scraper/lelo/cleaner.ts`
- 如实现中发现同类 prompt 只是在上述文件内重复出现，可一并按同原则收敛。

## 非目标
- 不修改 `function_tags` 的最终输出词。
- 不修改 `productKind`、`gender`、品牌过滤或商品分类逻辑。
- 不修改测试中为了覆盖业务场景而保留的显式输入文案。
- 不把天猫共用模板里的所有业务词都改成完全抽象的中性词。
- 不在本轮中重构 prompt 构造方式或抽公共 prompt 常量模块。

## 风险分级

### 高风险
以下词在角色设定或任务描述中更容易触发模型安全拦截，应优先替换：
- `情趣`
- `情趣用品`
- `情趣电商品牌`
- `情趣商品`

### 中风险
以下词未必直接触发拦截，但会增加模型把任务理解成敏感内容生成的概率，应同步收敛：
- `性健康装备`
- `玩法`
- `肛玩`
- `情趣玩具商品`
- 示例 JSON 中直接出现的 `G点刺激`

### 低风险
以下词与业务识别强相关，本轮默认保留，除非只存在于解释性 prompt 文案且替换不会影响模型理解：
- `安全套`
- `润滑液`
- `护理液`
- `吮吸`
- `玩具类商品`

## 替换策略

### 角色与任务描述
- `性健康装备选品专家` -> `个人护理设备选品助手`
- `情趣用品详情图识别助手` -> `个人护理用品详情图识别助手`
- `情趣商品数据清洗助手` -> `个人护理商品数据清洗助手`
- `情趣电商品牌数据清洗助手` -> `个人护理电商品牌数据清洗助手`
- `情趣硬件参数的数据拆解机器人` -> `个人护理设备参数的数据拆解助手`

### 功能和品类描述
- `玩法` -> `使用方式`
- `情趣玩具商品` -> `个人护理器具商品`
- `肛玩` -> `特定部位使用款` 或 `特定使用场景商品`
  - 具体选择以句子自然度优先，但必须避免重新引入其他高风险显式词。

### 示例 JSON 里的展示性标签
仅当这些标签出现在 prompt 示例中、且不是实际程序逻辑依赖值时，允许中性化：
- `G点刺激` -> `定点刺激`

注意：
- 如果某个词同时承担“模型理解提示”和“下游逻辑约束值”双重作用，则本轮不改。
- 所有替换都应保持原 prompt 的输出结构、字段名、字段类型和约束描述不变。

## 设计原则
- 只改 prompt 的自然语言说明，不改程序消费的结构化字段。
- 优先保证模型仍能正确完成商品识别、重排、清洗、OCR 摘取等任务。
- 中性化后仍需保留足够的业务边界，避免 prompt 过度抽象导致模型理解精度下降。
- 同一类角色描述在不同文件内尽量统一，避免出现多套中性称谓。

## 影响分析

### 推荐链路
`src/server/app-ai-service.ts` 与 `src/App.tsx` 中的推荐角色描述只承担“任务 framing”作用，不参与任何字段解析或分类判断，因此可直接中性化，不会影响结果对象结构。

### 官方站 cleaner
官方站 cleaner prompt 大多要求模型返回固定 JSON。这里最重要的是：
- 保持 JSON schema 不变
- 保持 `gender`、`material`、`price_*` 等字段要求不变
- 只替换描述性角色文案与少量展示性示例标签

因此这部分属于低结构风险修改。

### OCR prompt
OCR prompt 里如果把“玩法”改为“使用方式”，通常不会影响结构抽取能力，反而更贴近中性目录语境。需要避免把描述改得太泛，导致模型不知道应提取“模式与动力特征”这类字段。

## 实施方案

### 方案 A：仅做 prompt 文本定点替换（推荐）
- 只替换已识别的高/中风险词
- 不调整 prompt 结构
- 不抽象公共模板

优点：
- 改动最小
- 回归成本低
- 最符合本轮范围

缺点：
- 术语统一度只在局部改善，仓内仍会保留未处理的业务显式词

### 方案 B：在方案 A 基础上统一更多展示性示例词
- 除角色词外，再清理 prompt 示例 JSON 中更多感知上偏敏感的标签

优点：
- 中性化更彻底

缺点：
- 更容易让模型输出风格偏离现有标签习惯
- 需要更仔细地确认哪些词只是“示例”，哪些词已被下游暗含依赖

### 方案 C：全仓统一术语
- 连业务词表、标签、测试样例一起调整

优点：
- 表面上最统一

缺点：
- 风险明显超出本轮目标
- 极易影响分类、命中率、历史数据兼容和测试稳定性

本轮采用方案 A。

## 验收标准
- 所有本轮纳入范围的 prompt 文件中，高风险词 `情趣*` 不再出现在直接发送给模型的文案里。
- 推荐链路中的 `性健康装备选品专家` 全部替换为统一中性角色词。
- `玩法`、`肛玩`、`情趣玩具商品`、展示性 `G点刺激` 等中风险词在本轮纳入文件中完成替换或收敛。
- prompt 返回格式、字段名和代码消费逻辑保持不变。
- 现有相关测试至少完成针对性回归，确认没有因为字符串替换破坏结构化输出约束。

## 测试策略
- 运行与本次修改直接相关的测试：
  - `src/server/app-ai-service.test.ts`
  - 如存在官方站 cleaner 单测，则运行对应测试文件
- 对没有现成测试的 prompt 文件，至少做一次静态检视，确认：
  - 模板字符串仍合法闭合
  - JSON 示例仍合法
  - 关键字段约束未被误改

## 风险与回退
- 风险 1：中性化过度导致模型理解范围变宽，输出质量下降。
  - 控制方式：只替换角色和描述词，不删业务字段要求。
- 风险 2：误改了下游依赖的示例标签。
  - 控制方式：仅替换明确属于 prompt 说明层的展示词。
- 风险 3：前后端重复 prompt 未同步。
  - 控制方式：对 `src/server/app-ai-service.ts` 和 `src/App.tsx` 做同批次核对。

若回归发现输出质量异常，本轮改动应可通过回退单个 prompt 文案快速恢复，因为不涉及数据结构和业务流程调整。
