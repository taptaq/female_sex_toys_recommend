# 推荐结果页模型二次校准设计

## 目标
在推荐结果页中支持用户切换不同 AI 模型，并基于当前答题结果与当前候选池，对整套推荐结果进行一次“二次校准”。

这里的“整套结果”包括：
- Top 3 主推荐
- 备选卡片说明
- 选购建议

目标不是改变问卷逻辑或重新抓取产品，而是在既有结构化筛选和候选池基础上，让不同模型从不同视角重新组织最终推荐结果。

## 已确认决策
- 结果页支持展示当前全部可用模型，而不是只展示精选少量模型。
- 用户切换模型后，需要显式点击按钮触发重新校准，不做自动请求。
- 二次校准需要重算整个结果区，而不是只重排 Top 3。
- 默认首发推荐模型建议使用 `mimo-v2.5-free`，也就是当前 DMX 的 `dmxapi-mimo`。
- 结果页可以低调展示当前结果实际来源模型，主要用于观察调用命中情况，但不应过度工程化或破坏用户体验。

## 范围
- 修改 `src/pages/ResultsPage.tsx`，加入模型切换与重新校准交互模块。
- 修改 `src/App.tsx`，加入结果页二次校准所需的前端状态与调用流程。
- 修改 `src/server/index.ts`，新增统一的结果重校准接口。
- 根据需要补充模型展示配置与轻量状态辅助逻辑。

## 非目标
- 不改变问卷题目、答题流程和标签生成逻辑。
- 不改变装备库抓取、数据库内容或产品结构化打分规则。
- 不把结果页改造成多模型并排对比实验台。
- 不在本轮中实现“一次请求同时生成多套模型结果”。
- 不替换当前 AI provider ladder 的基础顺序；二次校准只是允许用户指定一个目标模型执行单次结果重算。

## 用户体验设计

### 模块位置
- 放在结果页顶部区域，位于“匹配完成”和标签摘要下方、主推荐卡片上方。
- 视觉上应与现有结果提示、主推荐卡片保持同一套玻璃拟态和青蓝色科技风格。
- 模块不应压过主推荐结果本身，重点是“提供一个再次校准入口”，不是变成主视觉。

### 模块内容
- 一段简短说明文案：
  - 例如“可切换不同模型，基于当前偏好与候选池重新校准整套推荐结果”
- 模型选择区：
  - 以可换行胶囊卡片或轻量按钮组展示全部可选模型
- 结果来源提示：
  - 例如“当前结果来源：Mimo（DMX）”
- 主按钮：
  - 默认文案可为“重新校准推荐”
  - 如果所选模型与当前结果模型一致，可显示“重新生成当前模型结果”
- 过程状态提示：
  - 例如“正在使用 Kimi（DMX）重新校准整套推荐...”

### 模型展示方式
结果页展示用户可理解的中文标签，同时保留低调的来源含义：
- `Mimo（DMX）`
- `Qwen（DMX）`
- `GLM（DMX）`
- `Kimi（DMX）`
- `DeepSeek（官方）`
- `Qwen（官方）`
- `GLM（官方）`

模型顺序遵循当前推荐链顺序，默认首选展示为：

`Mimo（DMX） -> Qwen（DMX） -> GLM（DMX） -> Kimi（DMX） -> DeepSeek（官方） -> Qwen（官方） -> GLM（官方）`

### 交互原则
- 默认选中当前结果实际来源模型。
- 用户切换模型后，不自动发请求，必须手动点击按钮确认。
- 重新校准时，禁用模型按钮与提交按钮，避免重复触发。
- 二次校准失败时，不清空已有结果，只提示失败，并保留上一版结果。
- 成功后一次性替换整个结果区，避免先替换 Top 3、再替换备选的碎裂感。

## 技术设计

### 总体策略
采用“后端统一编排，前端一次替换”的方案。

原因：
- 整套结果涉及两个现有 AI 阶段：
  - Top 3 重排
  - 备选说明与选购建议增强
- 如果前端自己编排两次请求，状态会更碎，失败处理也更复杂。
- 统一接口更利于后续记录模型命中、失败日志和调试信息。

### 新接口
新增统一接口：

`POST /api/ai/recalibrate-results`

### 请求体
请求体建议包含：
- `answers`
- `targetProvider`
- `targetModel`
- `filteredCount`
- `rerankPool`
- `backupCandidates`

更稳妥的实现建议是：
- 前端在本地继续负责结构化筛选、结构化打分和候选池构建
- 前端把当前已经形成的 `rerankPool` 和 `backupCandidates` 传给后端
- 后端只负责指定模型下的 AI 重排与增强生成

这样可以避免把结构化业务规则搬到服务端重复实现，也能保证“再次校准”和“初次推荐”面对的是同一批候选池。

### 返回结构
统一返回：

```json
{
  "data": {
    "topProducts": [],
    "backupProducts": [],
    "shoppingGuidance": [],
    "recommendationTips": []
  },
  "modelName": "mimo-v2.5-free",
  "provider": "dmxapi-mimo"
}
```

其中：
- `topProducts` 为完整可渲染的 Top 3 结果
- `backupProducts` 为补全说明后的备选卡片
- `shoppingGuidance` 为结果页选购建议
- `recommendationTips` 为当前候选紧张时的放宽条件建议
- `modelName` 与 `provider` 用于前端展示当前结果来源

### 服务端执行逻辑
新的 `/api/ai/recalibrate-results` 接口应：
1. 校验请求体
2. 仅使用用户指定的 provider/model 执行，不走整条兜底链
3. 基于传入 `rerankPool` 生成新的 Top 3 排序与推荐理由
4. 基于新的 Top 3 与传入 `backupCandidates` 生成新的备选说明和选购建议
5. 返回整套结果与本次实际命中的模型信息

### 与现有接口的关系
- 保留现有：
  - `/api/ai/rerank`
  - `/api/ai/result-enhancement`
- 新接口可以复用这些 prompt 构建思路，但由服务端在一个统一入口中串起来。
- 初次推荐流程可以暂时继续使用现有两段式接口，不要求本轮一并重构。

## 前端状态设计

### 新增状态
`src/App.tsx` 需要新增或整理以下状态：
- 当前结果来源 provider
- 当前结果来源 modelName
- 结果页当前选中的目标 provider
- 是否正在重新校准
- 重新校准错误信息

### 结果来源持久化
建议把以下内容纳入 `PersistedAppState`：
- `currentResultProvider`
- `currentResultModelName`

目的：
- 用户刷新结果页后仍能看到当前结果来源
- 模型切换 UI 能正确默认选中

### 调用时机
当用户在结果页点击“重新校准推荐”时：
1. 读取当前答案与当前候选池
2. 使用所选模型调用 `/api/ai/recalibrate-results`
3. 请求期间保留当前结果显示，同时展示顶部 loading 状态
4. 请求成功后一次性替换：
   - `topProducts`
   - `backupProducts`
   - `shoppingGuidance`
   - `recommendationTips`
   - 当前结果来源 provider/modelName
5. 请求失败后仅展示错误提示，不修改当前结果

## 默认模型策略
- 初次推荐默认仍优先使用当前主链首位：`dmxapi-mimo`
- 结果页模型选择默认选中当前结果实际来源
- 如果当前结果来源不可识别，则回退默认选中 `dmxapi-mimo`

选择 `mimo-v2.5-free` 作为默认推荐首发模型的理由：
- 更适合当前“在有限候选池中做中文选品重排与结构化输出”的任务形态
- 与当前 DMX 优先链的首位保持一致，产品逻辑统一
- 在推荐理由、备选说明和中文建议生成上更适合作为稳态默认模型

## 风险与约束
- 不同模型对 JSON 约束的稳定性可能不同，新的统一接口要延续当前的 JSON 解析与兜底保护。
- 二次校准属于增量操作，不能因为新请求失败而破坏当前已渲染结果。
- 模型列表展示全部开放后，用户可能频繁切换，因此前端需要强约束“切换不自动请求”。
- 如果未来 provider id 或模型名继续扩展，模型展示配置应集中维护，不要散落在页面 JSX 中。

## 验收标准
- 结果页可见“模型二次校准”模块，且移动端与 PC 都能正常显示。
- 用户可以切换全部当前可用模型，并显式点击按钮触发重新校准。
- 每次重新校准都会整套更新：
  - Top 3 主推荐
  - 备选卡片说明
  - 选购建议
- 当前结果来源模型可被低调展示。
- 校准失败时，旧结果保持不变，并向用户提示失败。
- 默认首发模型策略与当前主链首位保持一致，即 `dmxapi-mimo / mimo-v2.5-free`。
